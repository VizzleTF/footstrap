# Бенчмарк навигации: footstrap (SPA) vs bootstrap (полная перезагрузка)

Замеряет, насколько быстрее `luci-theme-footstrap` с клиентским SPA-роутером
(см. [docs/14](14-spa-router.md)) открывает страницы по сравнению со штатной
`luci-theme-bootstrap`, при обходе стандартных страниц меню по очереди с полным
рендером каждой.

Скрипт: `bench/nav-benchmark.py`. Воспроизводимо на любом роутере 25.12+.

## Что и как меряется

Для каждой страницы — время в миллисекундах (wall-clock) от «намерения
перейти» до «view полностью отрисован»:

- **footstrap** — реальный клик по пункту меню → SPA-роутер переинстанцирует
  view в `#view`, без перезагрузки страницы;
- **bootstrap** — полный переход по URL → браузер перезагружает весь каркас,
  заново парсит/исполняет `luci.js`+`cbi.js`, до-тягивает переводы, пересобирает
  меню, затем рендерит view.

Обе темы рендерят контент в **один и тот же** `#view` (его эмитит диспетчер,
независимо от темы), поэтому условие «отрисовано» одинаковое и сравнение
честное — меряется воспринимаемое пользователем время «клик → страница готова».

«Отрисовано» = в `#view` нет `.spinning`, есть дочерние элементы, нет
«Loading view…». Перед этим ждём появления спиннера — сигнал, что навигация
началась (иначе можно засчитать ещё старый контент).

### Метод (на каждую тему)

1. Активировать тему (`uci luci.main.mediaurlbase`), сбросить кэши LuCI.
2. Свежий контекст браузера, логин.
3. **Прогрев** — обойти все страницы один раз без замера (наполнить HTTP-кэш и
   кэш модулей LuCI → устойчивое состояние, «после того как всё прогрузилось»).
4. **Замеры** — `--runs` проходов (по умолчанию 5), берётся **медиана** по
   странице (глушит шум). Дополнительно считается число сетевых запросов за
   переход.

### Какие страницы

Определяются вживую из `/admin/menu`: только **листья-пункты меню** — узлы
`view` глубины `admin/<секция>/<страница>` (`len==3`) с путём, начинающимся на
`status/`, `system/` или `network/` (модули luci-mod-status / -system /
-network). Более глубокие узлы — это вкладки внутри view (`#tabmenu`, не клик по
меню) и в бенчмарк не входят. Сторонние приложения исключены по построению.

Страницы, требующие отсутствующего железа (Wi-Fi: `channel_analysis`,
`network/wireless`), виснут на спиннере в **обеих** темах и честно исключаются из
сравнения (логируется `! skip …`).

## Как повторить

```sh
# один раз: окружение (Playwright + Chromium)
python3 -m venv .venv
.venv/bin/pip install playwright
.venv/bin/python -m playwright install chromium

# запуск (тема автоматически возвращается к исходной в конце)
LUCI_PW=<пароль-root-роутера> .venv/bin/python bench/nav-benchmark.py \
    --ssh-host router --runs 5 --out bench/results-25.12.json
```

- `--ssh-host` — хост из `ssh -G` (по умолчанию `router`); роутер должен быть
  доступен по HTTP с машины запуска.
- `--runs N` — число замерных проходов (медиана). Меньше = быстрее, шумнее.
- `--out FILE` — сохранить результат в JSON.
- `--headful` — показать браузер (отладка).
- Скрипт **временно** переключает активную тему на роутере и в конце возвращает
  исходную (в т.ч. через `finally`). Если процесс убить `kill -9` — тема может
  остаться переключённой; вернуть вручную: `uci set luci.main.mediaurlbase=… ;
  uci commit luci`.

Время прогона ≈ `2 темы × (1 прогрев + runs) × N_страниц` навигаций — при
`runs=5`, 16 страницах это ~190 переходов, порядка 2–3 минут.

### Как читать

- **Сетевых запросов кардинально меньше**: 15–39 → 1–4. SPA тянет только нужный
  RPC; полная перезагрузка каждый раз перезапрашивает каркас + скрипты (пусть и
  304-ответами, но это раунд-трипы и повторный parse/exec).
- **Где выигрыш максимальный (3–7×)** — сетевые/init-bound страницы (network/*):
  экономится повторная инициализация рантайма и переводов.
- **Где выигрыш скромный** — Startup (1.13×) и Processes (1.47×): там время
  доминирует сам **рендер контента** (парсинг init-скриптов, таблица процессов),
  а не навигация. SPA убирает перезагрузку, но не ускоряет отрисовку самого view.

Абсолютные числа зависят от железа (тут — filogic); важны **отношения** и
падение числа запросов.
