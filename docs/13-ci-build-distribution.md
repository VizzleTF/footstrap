# CI, сборка пакетов и распространение

Как footstrap собирается в пакеты и раздаётся. Тема **noarch** (`LUCI_PKGARCH:=all`)
и поддерживает **OpenWrt 24.10** (`.ipk`/opkg) и **25.12+** (`.apk`/apk).

## GitHub Actions (`.github/workflows/build.yml`)

Триггеры: push в `main`, тег `v*`, любой `pull_request` или ручной `workflow_dispatch`.
Джобы: `check` → `lint` → `build` (`needs: [check, lint]`) → `release` (только на тег).

`permissions: contents: read` на уровне workflow; write объявляет только job `release`.
Раньше write-скоуп был workflow-wide, то есть выдавался каждому `pull_request`-прогону —
включая `npm ci` в lint, а значит lifecycle-скриптам всех dev-зависимостей.

### `check` — гейты без node (6 шагов)

Нужны только `sh`, `awk`, `python3`, `perl` и (для шаблонов) `cmake` — стоят секунды и не
могут сломать buildbot OpenWrt, где node нет и не будет.

1. **`sh -n`** по всем скриптам пакета, `install.sh`, `tools/*.sh` и `uci-defaults`.
   `tools/` в глобе потому, что `release-notes.sh` запускается **только** job'ом `release` —
   синтаксическая ошибка в нём обнаруживалась в момент, когда это дороже всего.
2. **Компиляция ucode-шаблонов.** Собирает `ucode` из коммита, запиненного в
   `luci-upstream.pin`, и гоняет `ucode -T -c` (это LuCI'шный trycompile) по каждому `.ut`.
   Импорты `luci.core`/`uci` — рантайм роутера, на раннере их нет, поэтому они **застаблены**
   (`-L`): шаблон реально парсится, тела стабов не исполняются (`-c` компилирует, не запускает).
   Гейт появился потому, что у шаблонов **не было парсера вообще**: luci.mk копирует `ucode/`
   verbatim, лишняя скобка в `header.ut` собиралась зелёной, релизилась — и у всех LuCI молча
   откатывался на другую тему (docs/01).
3. **ACL — валидный JSON.** rpcd **пропускает** нечитаемый файл в `acl.d` и молчит: лишняя
   запятая = грант не выдан никому, и больше ничто этого не замечает. Тема шипает ACL
   `luci-theme-footstrap.json` (это `uci footstrap` для «Save as default» + загрузка login-bg;
   `file.exec` на self-update теперь в отдельном репо `luci-app-footstrap-updater`, docs/23).
4. **`build-css.sh /tmp/cascade.css`** — броневой контроль: скрипт брейс-балансирует свой вывод
   и отказывается писать **подозрительно короткий** файл (обрезанная запись, сожранный хвост) —
   это **broken-build FLOOR** (80 КБ), корректностный гейт, **не бюджет размера**. Верхнего
   бюджета CSS больше нет (снят). Фактически сегодня `cascade.css` ~**111 КБ**; uhttpd отдаёт
   `/www/luci-static/*.css` **без сжатия** — это байты на проводе, но байты держатся суждением,
   а не потолком в CI.
5. **`audit.py --strict`** — undefined `var()`, затенённые декларации, чтение export-тира из
   `styles/`, мёртвые base-декларации, лишние `!important`, цветовые литералы.
6. **i18n**: `update-po.sh --check` — падает, если `.pot` устарел (строку добавили, скрипт не
   перезапустили) или хоть один `msgstr` пуст. Строка в `_()` без перевода рисуется
   по-английски **молча** — так весь поповер Appearance и оставался английским на русском LuCI.

### `lint` — npm-гейты

Живут **только** в CI: у buildbot'а нет node, и luci.mk он не нужен — тот копирует `htdocs/`
и `ucode/` как есть. Ничего из `package.json` **не шипается**. Локально всё сразу — `npm run
check` (полный список гейтов — в CLAUDE.md §«Lint / a11y gates»; таблица ниже — ключевые).

| Шаг | Что ловит |
|---|---|
| `eslint` (`htdocs`) | в т.ч. `wrap-regex` — запрещает `return /re/…`, форму, на которой jsmin ломается |
| `stylelint` (`styles/**`) | только correctness + инварианты проекта (не форматтер) |
| `a11y-gallery.mjs` | axe-core, WCAG 2.2 AA по `docs/gallery.html`, матрица {light,dark} × {footstrap,hicontrast} × {untinted,60°,260°} — **12 комбинаций** (переключатель палитры множит матрицу контраста) |
| `export-tier.mjs` | контракт `--*-color-*` с чужими `luci-app-*` — axe его не видит: их виджетов в галерее нет |
| `css-metrics.mjs` | храповик: `!important` ≤ порог, специфичность, пустые правила |
| `fs-orphans.mjs` | мёртвые `fs-*`-селекторы (безопасно только в нашем неймспейсе) |
| `css-dup.mjs` | одинаковые тела деклараций под разными гардами — их не поймает ни один линтер |
| `mirror.mjs` | `@mirror`-запиненные копии (CSS **и** shell) побайтово совпадают |
| `axes.mjs` | пре-пейнт в `head.ut` согласован с живыми аппликаторами Appearance |
| `chrome-fence.mjs` | метка `[data-fs-chrome]`, fence и pin всё ещё совпадают с хромом |
| `changelog.mjs` | контракт чейнджлога (docs/21): секции, порядок, зеркало RU, жирные лиды |
| `jsmin-verify.mjs` | **единственная** проверка, ловящая тихую порчу от jsmin (exit 0) — держит **исходник** jsmin-безопасным, ведь SDK/buildbot всё ещё гонит jsmin по неминифицированному источнику |

Про `jsmin-verify`: luci.mk минифицирует наш JS jsmin'ом на buildbot'е. jsmin решает,
`/` — регекс или деление, по **одному** предыдущему символу, и ни `n` (из `return`), ни `>`
(из `=>`) в allow-list не входят — `return /re/` заставляет его сожрать остаток файла **и
выйти с кодом 0** (openwrt/luci#8299, #8020, #8021, #8256). **Нулевой exit не доказывает
ничего.** Поэтому CI собирает тот же самый jsmin из `luci-base/src/jsmin.c` и доказывает, что
поток токенов вывода **идентичен** исходному (acorn). Список шипаемых JS не выписан руками —
`find htdocs -name '*.js'`: luci.mk копирует `htdocs/` целиком, значит это и есть шипаемое
множество.

**Минификация JS в CI идёт через terser, не jsmin.** Релизная сборка запускает
`tools/minify-js.mjs` (terser — умеет манглить идентификаторы, чего jsmin не может: ~41 КБ
против ~57 КБ у jsmin) и выставляет `FOOTSTRAP_PREMIN=1`, что переводит `LUCI_MINIFY_JS:=0` —
jsmin поверх terser-вывода снова открыл бы ловушку #8299 на формах, которые terser законно
эмитит. Сборка **без** этого шага (SDK-пользователь, buildbot — node нет) держит
`LUCI_MINIFY_JS` дефолтным `1`, и jsmin минифицирует нетронутый исходник как раньше. Поэтому
`jsmin-verify` и `wrap-regex` **обязательны для исходника** — фолбэк-путь всё ещё гоняет jsmin.
Численного бюджета минифицированного JS в CI больше **нет** (снят) — держим модули лёгкими
суждением, а не байтовым потолком.

### `luci-upstream.pin` — заимствованные тулы

Единственный источник запиненного коммита `openwrt/luci` **и sha256** двух инструментов,
которые CI **скачивает и ИСПОЛНЯЕТ**:

- `jsmin.c` — компилируется и решает, безопасен ли наш JS;
- `build/i18n-scan.pl` — запускается перлом (`update-po.sh`) и решает, полон ли каталог.

Взятые с движущегося `master`, эти гейты были бы «чем угодно, что упстрим запушил последним»;
sha256 говорит это вслух. Там же — `UCODE_PIN` (по коммиту, без sha256: имя git-объекта **и
есть** контрольная сумма его содержимого, а генерируемые GitHub'ом архивы побайтово
нестабильны) и **`USIGN_PIN` — коммит, из которого собирается роутерный `usign`**, чтобы
подписчик в CI и верификатор в поле были одним кодом. usign не требует ни cmake, ни libubox:
`cc *.c` по шести исходникам плюс bundled `base64.c`; собирает его `tools/build-usign.sh`,
который зовут **оба** нуждающихся job'а (`release` — подписать, `build` — верифицировать SDK),
потому что две копии `checkout <pin>` — ровно так два job'а оказываются на разных коммитах.
Коммит раньше был выписан дважды — в workflow и в `update-po.sh`, каждый с комментарием «bump
together» и ничем не подкреплённый.

### Матрица форматов (`build`)

| fmt | channel | SDK | CONFIG_USE_APK | ключ подписи SDK |
|---|---|---|---|---|
| apk | snapshot (25.12) | `snapshots/targets/x86/64` (apk-тулчейн 25.12+) | `y` | `b5043e70f9a75cde` (unattended-build) |
| ipk | 24.10 | последний `24.10.x` release SDK | `n` | `d310c6f2833e97f7` (release-ключ 24.10) |

Шаги каждой сборки:

1. Резолв версии: из тега (`v0.3.6` → `0.3.6`) или fallback `0.<date>.<run>`.
2. Скачать SDK и **верифицировать его ПОДПИСЬЮ.** URL резолвится **динамически** (grep по
   dir-листингу, не хардкод toolchain-версии; поддержаны `.tar.zst` и `.tar.xz`). SDK —
   наименее проверенный вход в репозитории и единственный, что попадает в пакет, который
   пользователи ставят, поэтому его проверяют **ed25519-подписью** `sha256sums.sig`, а не голым
   sha256 рядом. **sha256 рядом никогда не был проверкой:** кто может подменить
   `openwrt-sdk-*.tar.zst`, подменит и `sha256sums` рядом — тот же приём, что проект отвергает
   для GitHub-digest'а ассета (ниже). Подпись верифицируется публичным verify-ключом OpenWrt
   **для этой ветки**, скачанным из **`github.com/openwrt/keyring`** на коммите из
   `luci-upstream.pin` — *другой хост*, так что `downloads.openwrt.org` не ручается сам за себя.
   **Ветки не делят ключ:** 24.10 подписан своим release-ключом (`d310c6f2833e97f7`), 25.12 —
   unattended-build ключом (`b5043e70f9a75cde`), каждая нога называет свой в матрице рядом с
   `channel`; в пине лежит только *коммит* keyring'а. Затем sha256 из `sha256sums` (уже
   подписанного) сверяется как вторичный слой. Если сборка рапортует `BAD SIGNATURE` — прочти
   **keynum**, который печатает шаг, прежде чем думать про атаку: ротация ключа ветки OpenWrt
   выглядит идентично.
3. `rsync` каталога `luci-theme-footstrap/` в `package/luci-theme-footstrap/` SDK (сохраняя
   симлинки), `feeds update -a && install -a`.
4. `.config`: `CONFIG_PACKAGE_luci-theme-footstrap=m` (+ `CONFIG_USE_APK=y` для apk),
   `make defconfig`, `make package/…/compile V=s FOOTSTRAP_VERSION=<ver>`.
5. Ассерт, что в build-дереве есть `footstrap-theme.*.lmo`: каталог компилируется **внутрь**
   пакета (`Build/Prepare` → `po2lmo`), в git его нет, а молча пропавший `.lmo` = каждая `_()`
   рисует английский msgid и никто ни о чём не сообщает.
6. Собрать пакеты из `bin/` в `dist/`. Релиз несёт **два** пакета на формат — тему
   (`luci-theme-footstrap`) и апдейтер (`luci-app-footstrap-updater`) — и CI требует, чтобы
   **каждый** резолвился к **ровно одному** ассету под своим **name-anchored** regex
   (`luci-theme-footstrap[-_]…` / `luci-app-footstrap-updater[-_]…`). Это не гигиена, а защита
   старых роутеров: `footstrap-selfupdate.sh`, который у людей **уже стоит**, выбирает ассет по
   имени, а GitHub отдаёт ассеты **отсортированными по имени**. В 0.8.4 в релизе появился второй
   пакет (`luci-i18n-footstrap-ru`) — он сортируется раньше `luci-theme-footstrap-*`, и кнопка
   «Обновить» у всех ставила 6-КБ каталог вместо темы, рапортуя об успехе (issue #6). Именно
   поэтому апдейтер назван `luci-app-…`, а не `luci-theme-footstrap-updater` (последнее совпало
   бы с пиком темы и переоткрыло #6). Каталог переводов теперь **бандлится внутрь** темы (не
   отдельный `luci-i18n-…`), см. docs/02. **Апдейтер живёт в отдельном репозитории**
   (`VizzleTF/luci-app-footstrap-updater`) со своими CI/тегами/чейнджлогом — но релиз темы всё
   равно перечислен так, чтобы у обоих пакетов был точно один ассет (docs/23).
7. Job `release` (только на тег): скачивает артефакты обоих форматов, **подписывает каждый
   пакет** `usign`-ключом (секрет Actions; ключ строится `build-usign.sh` из `USIGN_PIN`) и
   кладёт `.sig`-ассет рядом, генерирует тело релиза из CHANGELOG (`tools/release-notes.sh` —
   жирный лид каждого пункта секции `## [x.y.z]`), `softprops/action-gh-release` цепляет пакеты
   и их подписи. Итог: один ассет темы на формат + его подпись (`.sig`) — подпись это то звено
   цепочки доверия, что переживает редирект (см. ниже).

Новый релиз: см. правила CHANGELOG в CLAUDE.md (переименовать `[Unreleased]` → коммит → тег
**этого** коммита), затем `git tag vX.Y.Z && git push origin vX.Y.Z` (пушить **на оба**
remote'а).

## Решения в Makefile (почему так)

Полный разбор — **docs/02**. Здесь только то, что касается сборки:

- **`LUCI_MINIFY_CSS:=0`** — критично. luci.mk гонит CSS через **csstidy**, а тот достаточно
  стар, чтобы манглить `:has()`, `color-mix()`, вложенный `calc()`: пакет ставится, верстка
  разваливается. Минифицирует вместо него `build-css.sh` (~111 КБ; верхнего бюджета нет, есть
  broken-build FLOOR).
- **`LUCI_MINIFY_JS` — два пути.** Релизная CI-сборка пре-минифицирует **terser**
  (`tools/minify-js.mjs`) и выставляет `FOOTSTRAP_PREMIN=1`, что переводит `LUCI_MINIFY_JS:=0`
  (~41 КБ; jsmin поверх terser переоткрыл бы #8299). Сборка **без** node (SDK-пользователь,
  buildbot) держит дефолтный `1`, и jsmin минифицирует нетронутый исходник (~57 КБ). Комментарии
  — ~60 % исходника JS, а uhttpd отдаёт `/www` без сжатия, поэтому минификация нужна на обоих
  путях. Безопасность на jsmin-пути обеспечивают `wrap-regex` и `jsmin-verify` (см. выше), а не
  отключение минификатора.
- **`$(TOPDIR)/feeds/luci/luci.mk`** (абсолютный) — относительный `../../luci.mk` работает
  только когда пакет лежит в `feeds/luci/themes/`; CI rsync'ает его в `package/<name>/`.
- **`LUCI_PKGARCH:=all`** — темы архитектурно-независимы; один пакет на все таргеты.

## Установка и цепочка доверия

Скрипт [`install.sh`](../install.sh) определяет `apk`/`opkg` и ставит **оба пакета** — тему и
`luci-app-footstrap-updater` — из последнего (или заданного) релиза, каждый по имени, из его
собственного репозитория (docs/23):

```sh
wget -qO- https://raw.githubusercontent.com/VizzleTF/luci-theme-footstrap/main/install.sh | sh
# закрепить версию:  ... | sh -s v0.3.6
```

Вручную — **raw-файл из релиза** (НЕ zip-артефакт из Actions):

```sh
apk add --allow-untrusted luci-theme-footstrap-*.apk   # 25.12+
opkg install luci-theme-footstrap_*.ipk                # 24.10
```

**`--allow-untrusted` = apk/opkg НЕ держит нашего ключа**, а не «байты не проверены». И
`install.sh` (его льют `curl | sh` **под root**), и `footstrap-selfupdate.sh` (бэкенд кнопки
«Обновить», за ACL rpcd, теперь в репо апдейтера) ставят пакет именно так — проверять его
**их собственная работа**, и цепочка доверия — **три** звена:

1. **Проверенный TLS-канал.** Никогда `curl -k`, никогда `--no-check-certificate` — и **не в
   качестве ретрая** после неудачной проверенной попытки: неудача **и есть** случай MITM, а
   `ca-bundle` входит в `DEFAULT_PACKAGES` OpenWrt, так что небезопасный путь не покупает
   ничего. Схема редиректа пинится (`--proto-redir '=https'`), но **только на curl-ветке**;
   `uclient-fetch` — пробуется первым и единственный на стоковом роутере — такого флага не
   имеет. Хост ассета пинится по allow-list'у, но **только на исходном запросе** — ассет релиза
   прыгает на `objects.githubusercontent.com`, и `-L` обязан пойти за ним. На пути, который
   стоковый роутер реально проходит, **подпись — единственный слой, переживающий редирект.**
2. **ed25519-подпись пакета** (`usign`) — звено, которое реально держит. sha256 в одиночку не
   стоит, и причина точна: GitHub **вычисляет** `@.assets[*].digest` из загруженных байт. Кто
   может подменить ассет релиза (утёкший write-PAT, без CI), тому digest пересчитают, и checksum
   проверит **пакет атакующего**. Ключ подписи — секрет GitHub Actions, его нет ни в одной ветке
   и не прочитать обратно — та же подмена валит подпись. Доказано на роутере end-to-end: ассет
   подменён + digest пересчитан → sha256 проходит, `ERR: BAD SIGNATURE`.
3. **sha256, который GitHub публикует для ассета** (`@.assets[*].digest`). Ниже подписи: ловит
   подменённую или обрезанную загрузку с **CDN ассетов** (другой хост, чем `api.github.com`) с
   более внятным отказом. Он **не** «остаётся, если usign нет» — не остаётся ничто: отсутствие
   usign = **отказ** (rc=2).

Всё падает **закрыто**: нет digest'а, нет `.sig`-ассета, нет usign на коробке — все отказывают.
У `install.sh` есть оверрайд (`FOOTSTRAP_ALLOW_UNVERIFIED=1`) для закрепления релиза старше
ключа; подпись, которая **есть и неверна**, не оверрайдится никогда. **Почему usign, а не
собственная подпись пакетника:** `apk verify` сверяет с `/etc/apk/keys` — доверять там ключу
footstrap значило бы сделать тему trust-anchor'ом для **всего**, что ставит роутер; opkg (24.10)
standalone `.ipk` вообще не верифицирует. usign есть на **каждом** образе OpenWrt (`base-files`
от него зависит), покрывает оба формата одним механизмом, и его ключ не авторизует ничего, кроме
этого пакета.

**`curl` НЕ входит в дефолтный набор пакетов OpenWrt** (базовый образ несёт `uclient-fetch`).
Self-updater поэтому откатывается на `uclient-fetch`, а не тянет curl зависимостью — благодаря
чему рантайм-список остаётся `+luci-base`. Раньше он жёстко требовал curl, и на стоковом
роутере кнопка «Обновить» умирала с «cannot reach the GitHub release API». `jsonfilter` и
`sha256sum` в базовом образе **есть**.

Два скрипта **не могут разделить файл** (installer — это `curl | sh`, он бежит **до** того,
как появится пакет, в котором лежала бы библиотека), поэтому их `fetch()`, allow-list хостов,
разбор ассетов и `verify_sig()` дублируются и пинятся `@mirror gh/{fetch,asset-host,asset-urls,
verify-sig}` побайтово (`tools/mirror.mjs`). Это не церемония: копии **уже** успели разойтись
тремя способами. **После сплита запиненная пара живёт в репо апдейтера**:
`footstrap-selfupdate.sh` уехал туда, а `@mirror`-группа с одной копией — жёсткий провал, поэтому
`install.sh` темы **снял маркеры `@mirror gh/*`** в том же коммите (`104d762`). Согласование
между двумя копиями `install.sh` (тема и апдейтер) ничем не гейтится — они вольны разойтись;
внутри каждого репо держатся только хелперы под маркерами.

Перевод едет внутри пакета темы (`/usr/lib/lua/luci/i18n/footstrap-theme.<lang>.lmo`) —
см. docs/02.

## Формат пакетов + типичная ошибка

- **apk** (25.12+) — формат Alpine apk-tools (OpenWrt перешёл на него в 25.12).
- **ipk** (24.10) — gzip-tar: `./debian-binary` (2.0) + `./control.tar.gz` + `./data.tar.gz`,
  ustar-заголовки. Идентичен формату родных OpenWrt-ipk.

**«Malformed package file» в opkg** почти всегда = ставят НЕ тот файл: zip-обёртку
GitHub-артефакта или `.apk` вместо `.ipk`.

Проверить `.ipk` настоящим opkg можно **вручную** (в CI такого прогона нет — сборка ipk
проверяется только тем, что SDK 24.10 её произвёл):

```sh
docker run --rm -v /path/pkg.ipk:/tmp/p.ipk openwrt/rootfs:x86-64-24.10.4 \
  sh -c 'mkdir -p /var/lock; opkg install --nodeps /tmp/p.ipk'
```
