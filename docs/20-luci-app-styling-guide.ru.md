# 20. Как стилизовать приложение LuCI, чтобы оно работало под любой темой

*English version: [20-luci-app-styling-guide.md](20-luci-app-styling-guide.md).*

Документ для авторов пакетов `luci-app-*`. Это не «правила footstrap»: каждое правило ниже выросло
из паттерна, найденного в живом популярном приложении, и каждое из них — баг **на какой-нибудь**
теме, обычно на той, против которой автор не тестировал. Там, где footstrap делает что-то своё, это
отмечено отдельно.

Что за этим стоит: 20 приложений, прочитанных построчно (OpenClash, passwall/passwall2, ssr-plus,
mosdns, vssr, ssclash, podkop, justclash, homeproxy, nikki, alist, dockerman, diskman, istore,
AdGuardHome, netspeedtest, partexp, taskplan и сток `openwrt/luci`), плюс самые звёздные
приложения, отрисованные на живом роутере. Фактура — в [19](19-tretya-storona-luci-app.md).

---

## 1. Время жизни CSS: куда положен `<style>`, тот и определяет, кого он заденет

Вьюха LuCI рисуется на клиенте и подменяется внутри `#view`. Современная тема может быть SPA — она
не перезагружает документ **никогда**. Поэтому у вопроса «когда мой CSS умрёт?» три разных ответа, и
безопасен только один.

**НАДО — класть `<style>` в дерево, которое вы возвращаете.** Он умрёт вместе с вьюхой, как и вся
остальная ваша разметка. Сток LuCI так и делает: `package-manager.js`, `nftables.js`, `aria2/log.js`.

```js
return E([], [
    E('style', [ css ]),           // живёт ровно столько, сколько вьюха — by construction
    E('div', { 'class': 'myapp-root' }, …)
]);
```

**НЕЛЬЗЯ — `document.head.appendChild(style)`.** На полной загрузке это выглядит безобидно: документ
всё равно выбросят на следующем клике. Под SPA-темой такой лист остаётся навсегда и перекрашивает
все страницы, куда пользователь пойдёт дальше. `luci-app-filemanager` именно так прячет стоковые
Save/Apply/Reset (`.cbi-button-save { display: none !important }`) — и любая конфиг-страница в этой
сессии становится несохраняемой. `luci-app-banip` и `luci-app-adblock` делают то же самое через
`<link>`.

**НЕЛЬЗЯ — инжектить на eval модуля.** `<style>`, добавленный на верхнем уровне модуля (или
библиотекой, которую вы грузите, — так делает ACE со своим 14-КБ `ace_editor.css`), создаётся **один
раз на документ** и повторно создан быть не может: модуль закэширован. Тема, которая приберётся за
вами, вернуть его не сможет, и виджет отрисуется без структуры. Если иначе никак — будьте хотя бы
идемпотентны и рассчитывайте, что лист живёт до перезагрузки страницы.

**Как реагирует footstrap** (чтобы это было предсказуемо). Не удаляется ничего и никогда. Лист в
`<head>` остаётся жить — и SPA-навигация продолжает работать — пока он не может покрасить не вашу
страницу:

* он называет только ваши классы (`.ace_*`, `.hexview`, `.myapp-card`), **или**
* он называет стоковый виджет, но **прикалывает** правило к вашей разметке собственным именем:
  `#cbi-myapp-section > .cbi-section-remove` и `.myapp-table th.sortable.active` — оба нормальны, без
  вашей секции или вашей таблицы они не сматчатся ни с чем. (Аргумент `:not()` пином не считается:
  `.cbi-button-save:not(.my-save-button)` по-прежнему матчит каждую стоковую кнопку Save на каждой
  странице.) **Или**
* это голый селектор, который не объявляет ничего, кроме ваших собственных кастомных свойств:
  `:root { --myapp-accent: … }` инертен, а `:root { color-scheme: light dark }` — нет.

Всё остальное — `.cbi-button-save { display: none }`, `.error { … }`, `h4 { … }`,
`div > label + select { … }`, `svg text { … }` — делает документ непригодным для SPA: следующий
переход выполнится полной загрузкой. Приложение продолжит работать, пользователь заплатит одной
обычной загрузкой. Все пять примеров — настоящие, из приложений топ-30 по звёздам.

## 2. Именуйте всё, что объявляете

Вы — гость в документе, который делите с темой и со всеми остальными приложениями.

- **Классы и id:** префикс по имени пакета — `.myapp-card`, `#cbi-myapp-…`. Не забирайте себе
  словарные слова. Встречено в живой природе, и каждое — реальная коллизия: `.hidden`, `.label`,
  `.description`, `.centered`, `.skeleton`, `.toast`, `.dialog-title`, `.flex-row`, `.log-line`, а
  также `[hidden] { display: none !important }` из `luci-lib-taskd`.
- **Кастомные свойства:** объявляйте на **своём** корневом элементе, а не на `:root`. `:root` —
  общая глобальная область: hex-редактор `luci-app-filemanager` кладёт туда `--clr-background`,
  `--clr-border`, `--clr-highlight`, а сам файловый менеджер — ещё и `color-scheme`. Тема, у которой
  токен случайно назван так же, молча перекрашивается. (OpenClash здесь как раз молодец: его ~90
  свойств объявлены на `.oc`.)
- **Никогда не везите глобальный ресет.** `* { margin: 0; padding: 0 }` (OpenClash — дважды)
  расплющивает хром самой темы. И CSS-фреймворк через `@import` (`pure.css` в vssr) — тоже: не вам
  ставить его на весь документ.
- **Никогда не перекрашивайте стоковые селекторы LuCI за пределами своей страницы.**
  `.cbi-button-up`, `.cbi-value`, `.cbi-section-table-titles`, `ul.dropdown`, `table th` —
  это владения темы, а ваше правило вне слоёв, то есть перебивает любой каскадный слой темы. Внутри
  своей страницы — скоупьте: `.myapp-root .cbi-value { … }`.

## 3. Цвет: берите экспортный тир, а не литералы

Любая тема LuCI отдаёт один и тот же контракт кастомных свойств. Читайте его — с литеральным
фолбэком:

```css
color:            var(--text-color-high, #333);
background-color: var(--background-color-low, #f5f5f5);
border-color:     var(--border-color-medium, #ddd);
/* статусные цвета: primary / success / warning / error, каждый в high / medium / low */
color:            var(--error-color-medium, #f44336);
/* у текста ПОВЕРХ заливки — своя краска: */
background-color: var(--success-color-high, #2e7d32);
color:            var(--on-success-color, #fff);
```

Правила:

- **Имена — точные.** `--warn-color-medium` (podkop, 7 использований) и `--text-color` (justclash,
  6) **не существуют** — они всегда падают в литерал, на любой теме. Правильно:
  `--warning-color-medium` и `--text-color-high`.
- **`high` / `medium` / `low` — это рампа**, а не три имени одного цвета. Если вы красите «нет
  данных» в `--primary-color-low`, ожидайте, что он будет заметно тише живого значения.
- **Не хардкодьте поверхность.** `background: white` на `position: fixed`-модалке (passwall) — это
  белый ящик на тёмной странице. `#1e1e1e` на лог-терминале (justclash) — чёрный ящик на светлой.
- **Никогда не кладите текст цвета C на полупрозрачный тинт того же C.** Тинт тянет фон к тексту и
  съедает собственный контраст, а будучи полупрозрачным — зависит от того, что под ним; безопасного
  процента не существует. Дайте чипу непрозрачную поверхность, а цвет пусть несёт рамка.
- `<font color="green">` (dockerman, ssr-plus, OpenClash) — это статус, который исчезает на половине
  палитр. Берите `--success-color-medium`.

## 4. Тёмная тема: `prefers-color-scheme` — не тот вопрос

`@media (prefers-color-scheme: dark)` сообщает про **операционную систему**, а не про тему. У любой
современной темы LuCI есть свой переключатель светлая/тёмная, и пользователь, включивший тёмную на
светлой ОС, получит вашу светлую карточку на своей тёмной странице. Этот баг есть у netspeedtest,
vssr, passwall и у стокового `ustreamer`.

Спрашивайте тему, в таком порядке:

```js
function isDark() {
    const root = document.documentElement;
    if (root.dataset.darkmode) return root.dataset.darkmode === 'true';  // так делают OpenClash и luci-app-internet-detector
    if (root.dataset.theme)    return root.dataset.theme === 'dark';     // footstrap, конвенция justclash
    if (root.dataset.bsTheme)  return root.dataset.bsTheme === 'dark';   // конвенция Bootstrap 5
    const m = getComputedStyle(document.body).backgroundColor.match(/\d+/g);   // работает на любой теме
    return m && m.length >= 3
        ? (0.299 * +m[0] + 0.587 * +m[1] + 0.114 * +m[2]) / 255 < 0.5
        : false;
}
```

Диалектов в природе четыре, и footstrap стампит три из них — `data-darkmode`, `data-theme`,
`data-bs-theme` — на `:root` до первой отрисовки, так что работает любая из проверок.
`luci-app-internet-detector` держит свой тёмный блок на `:root[data-darkmode="true"]` и поэтому
выглядит у нас правильно вообще без изменений.

То же самое в CSS:

```css
.myapp-card { background: var(--background-color-low, #fff); }        /* уже следует за темой */
:root[data-theme="dark"] .myapp-card { /* только если правда нужен тёмный оверрайд */ }
```

- footstrap стампит на `:root` и `data-darkmode`, и `data-theme`, и `data-bs-theme` — до первой
  отрисовки, ровно затем, чтобы работали все три диалекта.
- Яркостный фолбэк работает везде — но только потому, что тема держит фон `body` **непрозрачным**. Не
  рассчитывайте, что какой-то один атрибут есть всегда.
- **Лучше всего — не определять вовсе.** Если все ваши цвета взяты из экспортного тира, тёмная тема
  уже обработана — тир для этого и существует. `luci-app-nikki` не читает состояние темы вообще и
  выглядит правильно везде.

## 5. Редакторы и прочие тяжёлые виджеты

- **Не хардкодьте цветовую тему редактора.** mosdns и AdGuardHome прибивают CodeMirror к `dracula`,
  ssclash — ACE к `tomorrow_night_bright`: вечно чёрный редактор на светлой странице. Выбирайте тему
  редактора по `isDark()` выше (OpenClash делает это правильно: `themeExtension(isDark)`).
- **Не ставьте общую библиотеку в общий путь.** AdGuardHome кладёт свой CodeMirror в
  `/luci-static/resources/codemirror/`; два приложения с разными версиями затрут друг друга. Кладите
  в `/luci-static/resources/view/<yourapp>/`.
- **Iframe:** эталон — стоковый `luci-app-ttyd`: `width: 100%`, `min-height`, `resize: vertical`,
  порт из UCI, протокол зеркалит страницу. Не хардкодьте порт (dockerman: `:7682`) и не красьте
  iframe в `background: #fff` (netspeedtest) — встроенная страница сама решает, какого она цвета.
- **`z-index`:** оставайтесь в разумном диапазоне. Выпадашка passwall сидит на `2147483647`, а
  полноэкранный редактор OpenClash — на `999999 !important`; собственные оверлеи и дропдауны темы с
  этим сосуществовать не могут. Ничему из того, что вы рисуете, не нужно быть выше всего на свете.
- Никогда не делайте `body { pointer-events: none }` (passwall — на время своей модалки): если в этот
  момент произойдёт навигация, весь интерфейс останется мёртвым.

## 6. Раскладка: вьюпорт — это не ваша колонка

Тема ставит сайдбар рядом с вашей страницей. У footstrap это 224 px, и на экране 1024 px ваша колонка
контента шириной ~740 px, тогда как `@media (max-width: 768px)` всё ещё считает, что это «десктоп».
Все приложения из обзора, складывающие карточки по вьюпорт-запросу (justclash 56rem, podkop 900 px,
netspeedtest 768 px), ломаются ровно в этой полосе.

- Используйте `@container`-запросы или измеряйте свой контейнер и вешайте класс.
- Не сочетайте `table-layout: fixed` с `min-width` на каждой ячейке (filemanager) — это убивает любую
  адаптивную «карточную» раскладку темы.
- Избегайте фиксированных px-ширин у контейнеров; вы имели в виду `width: 100%; max-width: …`.

## 7. Дружите с SPA (на классической теме это ничего не стоит)

Современная тема может переинстанцировать вашу вьюху, ни разу не перезагрузив документ.

- `window.onload` и `DOMContentLoaded` срабатывают **один раз на документ**. Обработчик, повешенный
  там (diskman, `optimize_cbi_ui.htm` у passwall), после первой навигации молча перестаёт работать.
  Делайте работу в `render()`.
- Гасите свои `setInterval` и поллеры, когда вьюха уходит; регистрируйте поллеры через `L.Poll` — и
  LuCI сделает это за вас.
- Не добавляйте постоянные узлы в `document.body` (toast-контейнер podkop, диалог taskd): они
  переживают подмену вьюхи. Кладите их в дерево вьюхи.
- Не вешайте на документ `popstate`-обработчик, который навигирует (istore) — вы будете гоняться с
  роутером темы и выиграете, причём неправильно.
- **Не прячьте стоковые кнопки через CSS.** Поставьте `handleSave`, `handleSaveApply` и
  `handleReset` в `null` в своей вьюхе (так делают `luci-app-partexp` и `luci-app-ttyd`). Это
  поддерживаемый способ, ему не нужен `!important`, и он не утекает на следующую страницу.

## 8. Чек-лист

| | |
|---|---|
| `<style>` возвращается внутри дерева вьюхи, а не вешается в `<head>` | ✅ |
| Каждый класс, id и кастомное свойство с префиксом имени приложения | ✅ |
| Нет `:root { … }`, нет `* { … }`, нет импортированного CSS-фреймворка | ✅ |
| Ни один стоковый `.cbi-*` / `table` / `pre` не стилизован за пределами своего поддерева | ✅ |
| Все цвета из `--*-color-*` с литеральным фолбэком; имена правильные | ✅ |
| Тёмный режим — из `data-theme` / `data-bs-theme` / яркости body, но не из `prefers-color-scheme` | ✅ |
| Тема редактора выбирается по режиму страницы, а не захардкожена | ✅ |
| Раскладка привязана к контейнеру, а не к вьюпорту | ✅ |
| `handleSaveApply = null` вместо прятанья стоковых кнопок | ✅ |
| Нет `window.onload`, нет остатков в body, нет захвата `popstate`, нет `z-index: 2147483647` | ✅ |

Приложения, которые стоит читать как эталон: **`luci-app-nikki`** (ноль инжектов CSS, ноль литералов
цвета, только стоковые формы), стоковый **`luci-app-ttyd`** (правильно сделанный iframe),
**`luci-app-smartdns`** (его `<style>` возвращается внутри той самой `.cbi-map`, к которой относится)
и **`luci-app-internet-detector`** (кастомные свойства `:root { --app-id-* }` со своим префиксом и
тёмный блок по атрибуту темы).

## 9. Насколько всё это сломано на практике

Два круга обзора, 30 приложений, отрисованных на живом роутере под этой темой:

* **Большинство приложений в порядке.** Из следующей по звёздам десятки (quickfile,
  tailscale-community, smartdns, internet-detector, 3ginfo-lite, cloudflarespeedtest, aliddns,
  koolproxy, ech-workers, sms-tool) **все десять** сохраняют SPA-навигацию, и ни одно не инжектит
  ничего, чем можно покрасить чужую страницу.
* **Поломки сосредоточены в больших и старых**, и это почти всегда одни и те же две ошибки: CSS
  положен в `<head>` вместо дерева вьюхи, и цвет захардкожен вместо экспортного тира.
* **Легаси-приложения на Lua/CBI** (OpenClash, passwall, dockerman, diskman, AdGuardHome, koolproxy,
  sms-tool) рендерятся на сервере, поэтому их `<style>` живёт и умирает вместе с их собственной
  страницей — но захардкоженные светлые поверхности (карточки `#f7f7f7`, диалоги `background: white`)
  всё равно выглядят чужеродно на тёмной странице, и снаружи это не чинится ничем.
