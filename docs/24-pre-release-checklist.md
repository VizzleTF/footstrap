# 24 — Обязательный чек-лист перед релизом (реестр стражей закрытых issue)

Каждый закрытый баг оставил после себя **стража** — гейт или живую проверку, которая ловит его
повтор. Этот файл — единый список: строка на issue, механизм, команда, и автоматизирован ли он.
**Ни один релиз не выходит, пока раздел «Автоматические гейты» не зелёный, а раздел «Ручные живые
проверки» не пройден для тех issue, чей дифф релиза их задевает** (см. «Как определить объём» ниже).

Философия проекта: баг, у которого нет стража, повторится молча. Если чинишь то, чему здесь нет
строки, — заведи строку.

## Как определить объём (что вообще может регрессировать в этом релизе)

Регрессия приходит только оттуда, что дифф трогает. Перед проверкой:

```sh
git status --short | awk '{print $2}' | sed 's#/.*##' | sort -u
git status --short | grep -E 'styles/|menu-footstrap|\.ut$|fs-sheets|fs-select|fs-fit' \
  || echo 'no CSS / renderer / template / fence changes'
```

- Дифф **не трогает `styles/`, рендерер, шаблоны, fence** → визуальные и fence-issue
  (#1–#5, #7–#11) от этого диффа регрессировать не могут; достаточно того, что их гейты-стражи
  зелёные. Живой прогон не обязателен, но и не запрещён.
- Дифф трогает соответствующую область → живую проверку задетых issue провести обязательно.

**Всегда обязательно, независимо от диффа:** весь раздел «Автоматические гейты» + issue #6
(упаковка/выбор ассета) — потому что #6 ломается на уровне РЕЛИЗА, а не кода, и «ничего не
падает» ровно до момента, когда роутер в поле тянет обновление.

## Выбор ассета релиза — issue #6, отдельно и обязательно

Это самое хрупкое место КАЖДОГО релиза и единственное, что ломается тихо: release-ноты и выбор
ассета считаются в момент тега, в поле, месяцы спустя.

- **CI-гейт (`.github/workflows/build.yml`, job build):** ровно N ассетов на формат, каждый пакет
  резолвится своим **name-anchored** regex `^<name>[-_][^/]*\.EXT$` в ровно 1 штуку. Транзишн-релиз
  = 2 пакета/формат (тема + апдейтер); чисто-тема = 1. `.sig` оканчивается на `.EXT.sig` и под
  `\.EXT$` не попадает.
- **Полевой self-updater ≥ era name-matching** тянет тему `/luci-theme-footstrap[-_]…/` и апдейтер
  `/luci-app-footstrap-updater[-_]…/`, `head -1`. Проверить симуляцией на форме будущего релиза:

```sh
# на дев-роутере, RELJSON = json релиза нужной формы
for EXT in apk ipk; do for name in luci-theme-footstrap luci-app-footstrap-updater; do
  n=$(jsonfilter -i "$RELJSON" -e '@.assets[*].browser_download_url' | grep -Ec "/$name[-_][^/]*\.$EXT\$")
  echo "$name.$EXT -> n=$n (must be 0 or 1)"
done; done
```

- **Транзишн-специфика (0.9.4):** апдейтер собран в репе темы последний раз; `resolve_updater()`
  сначала репо апдейтера, потом релиз темы; версия на fallback — из ИМЕНИ ассета (`asset_ver()`);
  установленная == резолвнутая → скип без цикла переустановки. Первый тег репо апдейтера обязан быть
  **> версии транзишн-сборки** (opkg молча отказывает в даунгрейде). Проверено фикстурами на обоих
  форматах.

## Релизный ранбук (docs/21 — контракт чейнджлога)

Только после того, как всё выше пройдено:

1. `[Unreleased]` → `## [x.y.z] — YYYY-MM-DD` в **обоих** `CHANGELOG.md` и `CHANGELOG_ru.md`,
   добавить `compare/`-ссылку внизу. Секции в порядке Added/Changed/Deprecated/Removed/Fixed/
   Security/Performance, каждый bullet с `**bold lead**` (иначе выпадет из release-ноты).
2. `npm run changelog` — зелёный.
3. Коммит с этим изменением (сообщение в English, Conventional Commits, **без** AI-атрибуции).
4. Тег `vx.y.z` на **этот** коммит (никогда не тегать раньше записи).
5. Push тега и коммита в **оба** remote (`origin` + `forgejo`).
6. CI на `v*`: собирает оба формата, подписывает, `release-notes.sh` строит тело релиза из
   чейнджлога. Дождаться зелёного пайплайна; проверить, что в релиз попало ожидаемое число ассетов
   (+ `.sig` к каждому).
